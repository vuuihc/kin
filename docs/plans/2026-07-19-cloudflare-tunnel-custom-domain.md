# Cloudflare Tunnel + Custom Domain (Phone / Multi-device Remote)

**Status:** Design + ops implementation guide  
**Date:** 2026-07-19  
**Audience:** Maintainers and self-hosters who want a **stable, multi-device** public HTTPS path to a local Kin daemon  
**Related:** [docs/REMOTE_ACCESS.md](../REMOTE_ACCESS.md) · [docs/MVP_TECH_SPEC.md](../MVP_TECH_SPEC.md) §7.3 · [PRINCIPLE](../../PRINCIPLE.md) (no Kin cloud; BYO tunnel)

---

## 1. Outcome

After following this guide, a user can:

1. Keep `kin serve` (or Desktop daemon) bound to **loopback** `127.0.0.1:7777`.
2. Run a **named Cloudflare Tunnel** that terminates HTTPS on a **custom domain** (e.g. `https://kin.example.com`).
3. Open that URL on **phone, tablet, laptop, or any browser** (cellular or any network) with Kin token auth.
4. Approve tasks, stream transcripts, and open Artifacts against the **same** local daemon—no vendor agent cloud in the path.
5. Survive daemon/tunnel restarts with a **stable hostname** (unlike `*.trycloudflare.com` quick tunnels).

This is the recommended **public multi-device** rung when Tailscale/Funnel is slow, blocked, or undesirable. Kin remains BYO: **no Cloudflare SDK inside Kin** for v1.

---

## 2. Why this path (vs LAN / Tailscale / Funnel / frp)

| Path | Stable hostname | Multi-device without client app | Typical pain |
|------|-----------------|--------------------------------|--------------|
| LAN QR | No (LAN IP) | Same Wi‑Fi only | Leaves home/office network |
| Tailnet (tsnet) | MagicDNS name | Needs Tailscale on each client | Login friction; some regions unreliable |
| Funnel | Tailscale-provided HTTPS | Yes (browser) | Depends on Tailscale policy + account; not always available |
| frp + VPS | Yes (your domain) | Yes | You operate VPS + certs + reverse proxy |
| **Cloudflare Tunnel + custom domain** | **Yes** | **Yes (browser only)** | CF account + domain on CF DNS; free tier usually enough |

**Product fit**

- Aligns with “Kin is one HTTP port; any tunnel works” ([REMOTE_ACCESS.md](../REMOTE_ACCESS.md)).
- Does **not** introduce a Kin-hosted relay or Kin account.
- Custom domain + named tunnel is what makes daily phone use feel like UU-class “always the same address,” while still self-hosted at the origin.

**Explicit non-goals for this document**

- Embedding `cloudflared` inside the `kin` binary.
- Cloudflare Access / Zero Trust as a hard dependency (optional hardening only).
- Remote PTY / full desktop mirroring (terminal stays local-only per ADR 0004).
- Replacing Tailscale for users who already prefer a private tailnet.

## 3. Architecture

```text
Phone / laptop browser
        |
        | HTTPS (443)  https://kin.example.com/?token=…  or Bearer
        v
Cloudflare edge (TLS, CDN path for tunnel)
        |
        | Cloudflare Tunnel (outbound from home/office machine)
        v
cloudflared  (user process or service)
        |
        | HTTP  http://127.0.0.1:7777
        v
kin serve / Desktop-embedded daemon
  ├── token auth on every /api/* and UI
  ├── tasks, approvals, artifacts, settings
  └── /internal/* remains loopback-only (not exposed via tunnel if ingress is correct)
```

**Trust boundary**

- Origin daemon never opens a public inbound port if tunnel origin is loopback-only.
- **Application auth** = Kin bearer token (still required).
- **Transport** = Cloudflare-managed TLS to the edge; tunnel credential file is a second secret (machine identity to CF).

**WebSocket / streaming**

- Kin UI uses HTTP + WebSocket (or long-lived streams) on the same origin.
- Cloudflare Tunnel supports WebSockets on HTTP ingress by default; do **not** put an extra buffering proxy that strips `Upgrade` unless tested.
- If streaming stalls, check CF dashboard → tunnel metrics and local `cloudflared` logs before blaming Kin.

## 4. Prerequisites

| Item | Notes |
|------|--------|
| Kin daemon | `kin serve` (loopback) or Desktop that runs the same HTTP port |
| Domain | You control DNS; zone should be on Cloudflare (proxied names for tunnel routes) |
| Cloudflare account | Free tier is sufficient for personal use |
| `cloudflared` | Install from [Cloudflare docs](https://developers.cloudflare.com/cloudflare-one/connections/connect-apps/install-and-setup/installation/) |
| Token hygiene | Treat `~/.kin/token` as a password; prefer long random token already generated by Kin |

**Recommended origin bind:** loopback only.

```bash
kin serve
# listens on 127.0.0.1:7777 by default — prefer this behind cloudflared
```

Do **not** require `--lan` for Cloudflare; exposing `0.0.0.0` adds a second attack surface on the LAN with no benefit for the tunnel path.

## 5. Implementation (ops) — named tunnel + custom domain

This is the **stable** path. Prefer it over quick tunnels for daily multi-device use.

### 5.1 Install and login

```bash
# install cloudflared (macOS example)
brew install cloudflared

cloudflared tunnel login
# opens browser; authorize the domain/zone you will use
```

### 5.2 Create tunnel and DNS route

```bash
cloudflared tunnel create kin
# note TUNNEL_ID and credentials path printed (often ~/.cloudflared/<TUNNEL_ID>.json)

cloudflared tunnel route dns kin kin.example.com
# creates a CNAME kin.example.com → <TUNNEL_ID>.cfargotunnel.com (proxied)
```

Replace `kin.example.com` with your hostname. Subdomain is enough; root domain works but is rarely necessary.

### 5.3 Config file

Create `~/.cloudflared/config.yml` (or a path you will pass with `--config`):

```yaml
tunnel: <TUNNEL_ID>
credentials-file: /Users/<you>/.cloudflared/<TUNNEL_ID>.json

# Optional: reduce surprise if multiple configs exist
ingress:
  - hostname: kin.example.com
    service: http://127.0.0.1:7777
    originRequest:
      # Kin speaks plain HTTP on loopback; no TLS to origin
      noTLSVerify: false
      # Comfortable timeouts for long agent streams (tune if needed)
      connectTimeout: 30s
      # WebSocket / streaming friendliness
      disableChunkedEncoding: false
  # Catch-all required by cloudflared
  - service: http_status:404
```

**Rules**

- Only map hostnames you intend to expose.
- Catch-all `404` prevents accidental open routing.
- Origin must be the Kin port; do not point at unrelated local services on the same tunnel without separate hostnames and conscious risk acceptance.

### 5.4 Run the tunnel

Foreground (debug):

```bash
cloudflared tunnel --config ~/.cloudflared/config.yml run kin
```

### 5.5 Point Kin notifications / deep links at the public URL

Kin uses `ui.base_url` for Bark/ntfy deep links and “most public” connect hints.

```bash
# after auth: set stable public origin (no token in the setting)
curl -sS -X PUT "http://127.0.0.1:7777/api/settings" \
  -H "Authorization: Bearer $(cat ~/.kin/token)" \
  -H "Content-Type: application/json" \
  -d '{"ui.base_url":"https://kin.example.com"}'
```

Or Settings UI → `ui.base_url` = `https://kin.example.com`.

**Do not** store the Kin token inside `ui.base_url`. Token is supplied by the client (`Authorization` or first-load `?token=` then `localStorage`).

### 5.6 First phone open

1. On the phone browser (cellular is fine):  
   `https://kin.example.com/?token=<contents of ~/.kin/token>`
2. UI should load, strip token from the URL into `localStorage`, and show Tasks / Approvals.
3. Trigger an approval on the host machine; approve from the phone in one tap.

**Multi-device:** repeat the same URL on each device **once** (or paste token in Settings if the UI supports it). Each browser keeps its own stored token. No per-device Kin registration.

## 6. Quick tunnel (dev only — not the daily driver)

```bash
cloudflared tunnel --url http://127.0.0.1:7777
```

- Hostname like `https://random-words.trycloudflare.com` changes every process restart.
- Fine for a 10-minute demo; **unsuitable** for bookmarks, notifications, or multi-device habits.
- Still requires `?token=` / Bearer; still public to anyone who has the URL+token while the process lives.

## 7. Process supervision (stability is the product)

Named tunnel + custom domain only feels “UU-fast/reliable” if **both** `kin` and `cloudflared` auto-restart.

### 7.1 macOS `launchd` sketch (cloudflared)

`~/Library/LaunchAgents/com.user.cloudflared-kin.plist`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.user.cloudflared-kin</string>
  <key>ProgramArguments</key>
  <array>
    <string>/opt/homebrew/bin/cloudflared</string>
    <string>tunnel</string>
    <string>--config</string>
    <string>/Users/YOU/.cloudflared/config.yml</string>
    <string>run</string>
    <string>kin</string>
  </array>
  <key>RunAtLoad</key>
  <true/>
  <key>KeepAlive</key>
  <true/>
  <key>StandardOutPath</key>
  <string>/Users/YOU/.cloudflared/kin.out.log</string>
  <key>StandardErrorPath</key>
  <string>/Users/YOU/.cloudflared/kin.err.log</string>
</dict>
</plist>
```

```bash
launchctl load ~/Library/LaunchAgents/com.user.cloudflared-kin.plist
```

Run Kin Desktop (or a similar LaunchAgent/systemd unit for `kin serve`) so the origin is up when the tunnel is up. Order does not need to be perfect: cloudflared will retry origin connect.

### 7.2 Linux systemd sketch

`/etc/systemd/system/cloudflared-kin.service`:

```ini
[Unit]
Description=cloudflared tunnel for Kin
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=YOU
ExecStart=/usr/local/bin/cloudflared tunnel --config /home/YOU/.cloudflared/config.yml run kin
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl enable --now cloudflared-kin.service
```

### 7.3 Health checklist

| Check | Expect |
|-------|--------|
| `curl -sS -o /dev/null -w '%{http_code}\n' http://127.0.0.1:7777/` | `200` (or UI shell) locally |
| `curl -sS -o /dev/null -w '%{http_code}\n' https://kin.example.com/` | `401` or app shell without token; **not** CF 530/502 indefinitely |
| API with token | `curl -H "Authorization: Bearer $TOKEN" https://kin.example.com/api/tasks` → JSON |
| Phone Approvals | End-to-end approve while host on home network, phone on cellular |
| Sleep/wake | After laptop sleep, tunnel reconnects within ~1 minute without manual `run` |

## 8. Security model and hardening

### 8.1 Baseline (required)

1. **Token on every request** — public hostname makes the token the primary app barrier.
2. **Rotate on leak suspicion:** `kin token rotate` (daemon re-reads token file per request; no restart required).
3. **Credentials file** (`<TUNNEL_ID>.json`) is machine-level secret; do not commit to git; back up offline.
4. **Origin loopback only** — avoid `--lan` unless you also need same-Wi‑Fi QR without CF.
5. **`/internal/*`** must not be published as a separate ingress; default single-origin Kin port is OK because those routes stay loopback-restricted in Kin in addition to token auth (verify after upgrades).

### 8.2 Optional: Cloudflare Access (Zero Trust)

For higher assurance (family domain, shared Wi‑Fi fears, token-in-URL shoulder surfing):

1. Cloudflare Zero Trust → Access → Application → Self-hosted: `kin.example.com`.
2. Policy: allow your email / IdP / one-time PIN.
3. Users pass Access **then** still need Kin token (defense in depth).

Trade-off: every browser must complete Access; slightly more friction than token-only. Good for “I pasted a token in a screenshot once.”

### 8.3 Optional: harden token URL usage

- Prefer pasting token once into a trusted bookmarklet or password manager note; avoid chat apps.
- After first load, confirm address bar no longer contains `token=`.
- If a tokenized URL may have leaked: rotate immediately.

### 8.4 What we will not claim

- Cloudflare cannot read TLS at the edge for tunnel traffic in the usual Tunnel model the same way a random reverse proxy might—but **you still chose a third-party edge**. Threat model = personal productivity console, not air-gapped secrets store.
- Token-only public apps are never “anonymous-safe.” Treat the hostname as sensitive infrastructure.

## 9. Multi-device behavior (product expectations)

| Concern | Behavior |
|---------|----------|
| Device count | Unlimited browsers; no Kin device registry |
| Auth | Same shared token (or rotate + re-provision all devices) |
| Concurrent use | Multiple clients against one daemon; task/approval state is server-authoritative |
| Offline host | Tunnel edge up but origin down → 502/504; phone should show reconnect/error, not empty success |
| Desktop + phone | Desktop may talk to `127.0.0.1`; phone uses `https://kin.example.com`; same SQLite/data dir |

**No Kin code change is required** for multi-device once the tunnel and `ui.base_url` are set. Optional UX improvements are §11.

## 10. Kin-side settings map

| Setting / artifact | Role |
|--------------------|------|
| `~/.kin/token` | Shared secret for all clients |
| `ui.base_url` | Public origin for notification deep links (`https://kin.example.com`) |
| `notify.bark_url` / `notify.ntfy_topic` | Push on approval/task complete; links should open public host |
| Settings → connect URL / QR | Today often reflects active listeners; for CF-only loopback, **manually** share `https://kin.example.com/?token=…` or improve UI (§11) |

If both Funnel and Cloudflare are configured, pick **one** public URL for `ui.base_url` to avoid confusing deep links.

## 12. Acceptance criteria (this guide)

- [ ] Named tunnel + `kin.example.com` serves Kin UI over HTTPS from cellular.
- [ ] Token required; unauthenticated API calls fail.
- [ ] `kin token rotate` invalidates old phone sessions without daemon restart; new token re-provision works.
- [ ] Approval request on host → phone Approvals → Approve → task proceeds.
- [ ] Transcript streaming usable on phone (no immediate WS failure).
- [ ] Artifacts library opens on phone via same host (read path).
- [ ] `ui.base_url` set; test notification deep link opens public host (if Bark/ntfy configured).
- [ ] `cloudflared` under launchd/systemd reconnects after reboot without manual steps.
- [ ] Quick tunnel documented as **dev-only**; named tunnel is the recommended daily path.

## 13. Failure modes and fixes

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| CF 502/504 | `kin` not listening on 7777 | Start daemon; verify `curl 127.0.0.1:7777` |
| CF 530 | Tunnel not connected | `cloudflared tunnel run`; check credentials path |
| UI loads, API 401 | Token missing/stale | Re-open with `?token=`; or rotate + update all devices |
| Works on Wi‑Fi not cellular | Hit LAN IP by mistake | Use `https://kin.example.com`, not `192.168…` |
| Notifications open wrong host | `ui.base_url` still loopback/LAN | Set to `https://kin.example.com` |
| Random hostname every day | Using quick tunnel | Migrate to named tunnel §5 |
| WS drops often | Aggressive client network; sleep | KeepAlive on cloudflared; retest on stable cellular |

## 15. Recommended default for dogfood

For maintainers who want “phone always works, multi-device, stable URL”:

1. `kin serve` on loopback (or Desktop equivalent).  
2. Named Cloudflare Tunnel + custom domain (§5).  
3. launchd/systemd for `cloudflared` (§7).  
4. `ui.base_url=https://kin.example.com`.  
5. Bark or ntfy for approval push.  
6. Keep Tailscale/Funnel optional for private-only access; do not require both.

This is the practical answer to “faster/more stable than Claude/Codex remote for multi-device supervision” without building a Kin cloud.
