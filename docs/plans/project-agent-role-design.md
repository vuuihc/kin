# Kin 在项目中的角色与教练循环

**状态：** Proposed（Project P1/P2 的交互设计，不代表已实现）

**日期：** 2026-07-22

**关联：** [ADR 0008](../adr/0008-project-one-pager.md) · [Project + One-Pager 计划](./project-one-pager.md) · [TODO](../TODO.md) · [ADR 0003（Artifacts）](../adr/0003-artifacts-and-reader.md) · [PRINCIPLE §5.7 / §5.11](../../PRINCIPLE.md)

## 1. 决策摘要

Kin 在一个项目里仍然是**同一个 Kin**，不增加一组需要用户管理的角色或 Agent。项目模式只改变 Kin 的工作姿态、提问方式和 One-Pager 模板：

| 项目模式 | Kin 的默认姿态 | 优先帮助用户回答 |
|---|---|---|
| Ship | 交付搭档 | 什么结果算交付？当前最短主线是什么？ |
| Learn | 教练 | 用户能否解释、练习并迁移所学？ |
| Explore | 研究搭档 | 哪个假设值得验证？什么证据会改变判断？ |
| Maintain | 守护者 | 如何低风险地恢复健康并留下可复用知识？ |

姿态不是权限、人格或独立线程。执行任务的主 Agent 始终围绕用户确认的 **North Star** 与 **Current Focus** 干活；Kin 的教练循环只在合适的边界提出少量问题和补丁，不与主任务争夺注意力。

## 2. 要解决的问题

One-Pager 能回答“这个项目是什么、现在做什么”，但仅有静态模板仍可能变成一张很快过期的封面。反过来，若 Kin 持续打断、自动总结或维护大量状态，项目又会退化成看板和汇报系统。

需要的是一个轻量闭环：

```text
对齐目标 → 围绕 Focus 执行 → 看证据 → 短复盘 → 提议 0～3 条补丁 → 用户决定
    ↑                                                        │
    └──────────────── 下一个 session 继续 ────────────────────┘
```

这个闭环必须同时满足：

1. 主任务优先，教练行为不能成为前置仪式。
2. 用户拥有目标与判断，Kin 不偷改项目叙事。
3. One-Pager 保持短而有用，Artifacts 保存展开后的成果和证据。
4. 不同模式有不同追问，但共用一套产品对象和交互。

## 3. 单一身份，三种职责

这些职责是同一个 Kin 在一次项目会话中的不同时刻，不是三个可选择的 Agent。

### 3.1 执行搭档（默认、前台）

- 读取短项目上下文：North Star、Current Focus、最多 3 条 Next、必要的证据指针。
- 把用户本次请求作为直接目标；若它与 Focus 不一致，只在偏离会明显浪费工作时提醒一次。
- 计划、调用工具、产出代码或内容；权限仍由现有审批体系决定。
- 不为了“完善 One-Pager”暂停可以直接完成的工作。

### 3.2 教练（按边界触发、低频）

- 在目标含糊、方向分叉、证据与假设冲突或用户主动复盘时提问。
- 一次只解决当前决策，不进行通用式盘问。
- Learn 强调复述与迁移，Explore 强调可证伪假设，Ship 强调验收与风险，Maintain 强调回归与可恢复性。
- 用户可以跳过；跳过不阻塞任务。

### 3.3 副编辑（收工、后台提议）

- 从本次 session 的真实结果中提取候选结论、未决问题和下一步。
- 最多提出 0～3 条 One-Pager 补丁；没有高信号内容时返回 0 条。
- 为补丁附上来源 session、Artifact 或文件指针。
- 只提议，不直接改写 North Star、Focus 或用户正文。

## 4. 模式契约

模式决定关注点，不决定固定流程。用户随时可以切换；切换只更新模板提示和后续建议，不重写已有内容。

### 4.1 Ship — 交付搭档

关注：可演示的完成定义、当前关键路径、外部约束、回归风险。

- 开始时：若缺失，只追问一个足以开工的验收边界。
- 执行时：优先完成可验证的纵向切片。
- 收工时：记录已验证结论、剩余阻塞和下一个最短动作。
- 避免：虚构完成百分比、为了里程碑而制造流程。

### 4.2 Learn — 教练

关注：理解是否能被用户复述、应用和迁移，而非阅读了多少材料。

- 开始时：确认用户想“知道”“能解释”还是“能独立完成”。
- 执行时：先给脚手架；在合适节点让用户预测、解释或尝试。
- 收工时：优先提议 Teach-back、仍模糊处和一个练习。
- 展开的讲义、例题和测验保存为 Artifact；One-Pager 只留能力目标与短结论。
- 避免：每一步都提问、把聊天变成考试、用 streak 制造压力。

### 4.3 Explore — 研究搭档

关注：假设、证据、反例、停止条件和下一次最便宜的验证。

- 开始时：把宽泛主题缩成当前要降低的不确定性。
- 执行时：区分事实、推断与偏好；保留被否定路径的理由。
- 收工时：更新最可信判断、仍开放的问题与下一项实验。
- 深度调研或模块图保存为 Artifact，并从 One-Pager 链回。
- 避免：过早收敛成路线图，或把所有搜索结果堆进封面。

### 4.4 Maintain — 守护者

关注：健康基线、已知风险、回归保护、恢复路径和 do-not-touch 区域。

- 开始时：确认故障/维护窗口、风险容忍和可回退点。
- 执行时：优先诊断与最小可逆变更，保留验证证据。
- 收工时：记录根因、回归测试、操作陷阱和后续健康动作。
- Runbook、事故复盘等长文保存为 Artifact。
- 避免：借维护任务发起无关重构。

## 5. 会话生命周期

### 5.1 新会话：封面即入口

新会话页展示当前 cwd 对应的 One-Pager 摘要，替代项目无关的示例提示。用户仍可直接输入任何任务；尚未物化项目时不强迫创建。

创建项目任务时注入有上限的短上下文：

```text
Project / mode
North Star
Current Focus
Next（最多 3 条）
必要的 session / artifact 指针（最多 3 条）
```

完整 One-Pager 不默认进入每次模型上下文。若用户请求与 Focus 明显无关，Kin 可以问“这次是临时支线，还是要更新 Focus？”，但用户不回答也可以继续。

### 5.2 执行中：主 Agent 干活

主 Agent 以本次请求和 Current Focus 为工作约束。教练介入仅限四种高价值信号：

1. 缺少一个会导致方向完全不同的决定；
2. 出现两个以上代价显著不同的方向；
3. 新证据推翻 One-Pager 中的关键假设；
4. 用户主动要求解释、复盘或挑战判断。

普通实现细节、可安全推断的选择和可逆步骤由 Agent 直接推进。

### 5.3 收工：短复盘与补丁

终止较长任务、存在实质结果或用户点击“收工”时，生成一张非阻塞卡片：

1. 一句话说明本次得到什么；
2. 0～3 条候选补丁，分别落到 Conclusions / Open Questions / Next；
3. 可选 Focus 变更，单独呈现；
4. 每条给出依据。

每条补丁支持 **采纳 / 编辑后采纳 / 忽略**。North Star 永不作为批量补丁静默修改；若 Kin 判断目标已漂移，只提出一个单独问题。

### 5.4 再进入：继续，而非重新汇报

下次打开项目时优先展示 Current Focus、最近一次有效变化和最多 3 条 Next。Catch-up 只在封面可能过期或用户主动触发时运行，并始终生成 diff。

## 6. One-Pager 与 Artifacts 的边界

| 内容 | 去向 | 理由 |
|---|---|---|
| North Star、Current Focus、短结论、开放问题、Next | One-Pager | 一眼恢复项目方向 |
| 教程、研究报告、设计稿、复盘、runbook、模块图 | Artifact | 内容需要独立阅读、引用和保留来源 |
| 执行过程、工具输出、尝试细节 | Session | 可审计过程，不污染项目封面 |
| 稳定的个人偏好或跨项目事实 | Governed Memory（以后） | 不应误写成项目状态 |

判断规则：如果一项内容无法在一两行内帮助“下一次继续”，就不应直接塞进 One-Pager；可以保存为 Artifact，再在 Evidence 中留链接。

## 7. 建议与写入策略

每条候选补丁使用稳定结构，便于 UI 渲染和审计：

```text
target: conclusions | open_questions | next | focus
operation: append | replace
text: ...
reason: ...
evidence: [task_id / artifact_id / file]
confidence: low | medium | high
```

约束：

- 一次 0～3 条；Focus 建议单列，不占用普通补丁后偷偷合并。
- 仅从已发生、可指向的结果提议，不把模型猜测写成事实。
- 相同或语义重复内容不再提议。
- 用户忽略的建议不在没有新证据时重复出现。
- 采纳写入使用现有 optimistic concurrency；冲突时重新生成 diff，不覆盖新版本。
- 失败不影响任务完成状态，建议可稍后处理或丢弃。

## 8. 产品表面

P1 只需要三个轻量入口：

1. 新会话页的 One-Pager 摘要与“打开封面”；
2. session 结束后的补丁卡片；
3. Project Home 的 Catch-up / 待审建议。

不要新增“角色选择器”、角色头像、教练专属导航、每日打卡或独立教练会话。P2 的 One-Pager Companion 仍是同一个 Kin，只是上下文范围固定在该项目封面及用户明确选中的证据。

## 9. 权限、失败与信任

- 教练姿态不提升工具权限；所有外部副作用继续走现有审批和审计。
- 自动读取只限当前项目已关联的 One-Pager、session 摘要和显式证据；不借 Catch-up 扫描无关目录。
- LLM 不可用时，任务照常完成；可用确定性信号生成空建议或简单证据列表，不伪造叙事。
- 建议生成、采纳人与最终 diff 应可追溯。
- 用户可以关闭收工建议，或针对项目关闭教练提示。

## 10. 分期

### P1 — 先闭合回收循环

- 结构化 0～3 条建议与 diff 审核；
- Ship / Learn / Explore / Maintain 的提示策略；
- 手动“收工”和高信号自动触发；
- Catch-up、去重、忽略记忆与并发保护；
- One-Pager / Artifact 目标分流。

### P2 — 深化上下文，不增加角色实体

- One-Pager Companion（复用 Artifact companion 的线程模型）；
- 更精确的 Focus 注入和 evidence packing；
- 跨 Agent handoff pack；
- 根据真实使用数据调整模式提示。

## 11. 验收标准

1. 用户不理解“教练循环”这个术语，也能直接从新会话页开始工作。
2. 一个有实质结果的项目 session 能在约 20 秒内审完 0～3 条补丁。
3. 未经确认，North Star、Focus 和 One-Pager 用户正文不会改变。
4. Learn/Explore/Maintain/Ship 的建议侧重点不同，但 API、存储与主 UI 不增加角色实体。
5. 长内容进入 Artifact，One-Pager 仍可在一屏附近恢复方向。
6. 用户跳过复盘、模型失败或项目不存在时，普通 task 流程零回归。
7. 建议包含可访问的来源；无法提供依据时不生成事实性结论。

## 12. 非目标

- 面向用户的多 Agent 角色编排；
- 强制苏格拉底式问答或每步确认；
- 自动改写 North Star；
- 项目完成百分比、连续天数、红黄绿健康分；
- 把 One-Pager 变成长报告或全量记忆库；
- 在 P1 引入新的通用工作流引擎。
