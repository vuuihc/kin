# Kin 产品指导纲领

[English](./PRINCIPLE.md)


**状态：** Draft v0.1
**项目性质：** 开源、Local-first、跨设备、模型无关的个人 Agent
**名称：** Kin
**产品主张：**

> Your agent. Your memory. Any model.

中文表述：

> 属于你的个人 Agent。保留你的记忆和习惯，不依附于任何模型厂商。

---

## 1. 产品定义

Kin 是一个由用户拥有的、跨设备连续存在的个人 Agent。

它能够长期理解用户的背景、目标、偏好和做事习惯，与用户讨论问题、提供建议，并在获得适当授权后，调用工具完成实际任务。

Kin 不等同于某一个大语言模型。

Claude、GPT、Gemini、DeepSeek、本地模型及其他推理服务，只是 Kin 在不同任务中可以使用的认知后端。用户可以更换模型，而不失去 Kin 的记忆、人格、配置、工具、工作流和历史关系。

可以用以下模型理解 Kin：

```text
Kin
├── Identity       它是谁，以及它与用户的关系
├── Memory         它对用户和过去经历的理解
├── Habits         用户形成的习惯、偏好和工作方式
├── Agency         它可以规划和执行哪些事情
├── Tools          它被允许调用的系统和服务
├── Policies       权限、确认、安全和隐私规则
└── Embodiments    macOS、iOS、Android、Web、CLI 等不同载体
```

模型是 Kin 可以更换的“大脑”。

客户端是 Kin 在不同设备上的“身体”。

工具是 Kin 用来作用于现实世界的“手”。

记忆、身份和用户关系才是 Kin 的连续主体。

---

## 2. 愿景

Kin 的长期目标不是成为一个功能更多的聊天客户端，而是成为用户长期拥有的个人智能基础设施。

一个成熟的 Kin 应当具备以下特征：

1. 用户不需要在每次对话中重新解释自己。
2. 用户更换设备后，面对的仍然是同一个 Kin。
3. 用户更换模型服务后，Kin 的身份和长期记忆不会丢失。
4. Kin 能够理解用户惯用的表达、工作方式和判断标准。
5. Kin 不只是回答问题，也能在清晰的权限边界内执行任务。
6. 用户能够检查、修正、导出和删除 Kin 保存的任何个人信息。
7. Kin 可以逐渐成为用户能力的延伸，但永远不取代用户对重要决定的控制。

长期来看，Kin 应当接近：

> 一个了解我、能够与我商量问题、可以代表我处理部分事务，同时始终由我控制的数字伙伴。

---

## 3. 要解决的问题

现有 AI 产品通常存在以下结构性问题。

### 3.1 模型与用户身份被绑定

用户对模型的使用习惯、历史上下文和长期记忆，经常被锁定在某一家模型厂商或某一个客户端中。

切换模型意味着：

* 重新配置 Prompt；
* 重新建立上下文；
* 丢失个人记忆；
* 丢失工具和工作流；
* 重新培养交互习惯。

Kin 要把个人 Agent 的身份从模型服务中分离出来。

### 3.2 服务端优先，客户端只是入口

许多 Agent 系统以 Server、Gateway 或 Bot Channel 为中心，客户端只是服务器能力的一个薄壳。

这种架构容易忽视：

* 原生快捷键；
* 文件和剪贴板；
* 分享菜单；
* 系统通知；
* 移动端意图；
* 设备传感器；
* 离线能力；
* 用户在不同设备上的具体使用习惯。

Kin 必须坚持 Client-first。每一个客户端都不是简单的聊天窗口，而是 Kin 使用该设备能力的原生化身。

### 3.3 长期记忆不可见、不可控

许多产品声称“记住用户”，但用户无法清楚知道：

* 系统记住了什么；
* 为什么记住；
* 信息来自哪里；
* 当前是否仍然准确；
* 是否存在冲突；
* 如何修改和删除。

Kin 的记忆必须可检查、可解释、可修正、可迁移。

### 3.4 Agent 能做事，但权限边界模糊

Agent 一旦具备文件、邮件、日历、浏览器、终端和账户访问能力，就可能造成真实世界影响。

Kin 不应把“自主程度高”简单等同于“产品更先进”。

Kin 的目标是：

> 在尽可能低的操作摩擦下，保持行为可理解、可授权、可审计、可撤销。

---

## 4. 目标用户

Kin 首先服务于愿意长期使用、配置并拥有自己 Agent 的个人用户。

核心用户包括：

* 开发者和工程师；
* 独立开发者；
* 研究人员；
* 知识工作者；
* 重视隐私和数据主权的 AI 重度用户；
* 同时使用多个模型和多个设备的用户；
* 希望逐步建立个人自动化工作流的用户。

Kin 初期不以“完全不配置的普通消费者”为主要目标。

早期产品可以具有一定技术门槛，但必须做到概念清晰、默认配置可靠、日常使用简洁。

---

## 5. 核心产品原则

### 5.1 用户拥有 Kin

Kin 的个人数据、记忆、配置和工作流属于用户，而不是项目维护者、云服务商或模型厂商。

必须支持：

* 本地使用；
* 数据导出；
* 数据删除；
* 自托管；
* 更换同步服务；
* 更换模型供应商；
* 在不使用官方服务的情况下保持核心功能可用。

Kin 不得把官方云账户设计为核心功能的强制依赖。

---

### 5.2 Local-first

用户的本地设备是 Kin 数据的主要归属地，而不是云端缓存。

Local-first 不等于完全拒绝云服务，而是意味着：

* 本地数据是第一份数据；
* 离线时仍能访问已有信息；
* 云端只负责用户明确启用的同步、备份或远程执行；
* 用户可以知道哪些数据离开了设备；
* 敏感凭据默认不跨设备同步；
* 云端服务应当可以被替换或自托管。

---

### 5.3 模型无关

Kin 的核心能力不得依赖某一家模型独有的 API 语义。

模型后端应通过统一能力层接入，例如：

```text
Generation
Streaming
Structured Output
Tool Calling
Vision
Audio
Reasoning
Embeddings
Reranking
Context Caching
```

不同模型能力可以不同，但 Provider 特性不应污染 Kin 的身份、记忆和工具层。

用户应当能够：

* 为不同任务指定不同模型；
* 设置默认模型和备用模型；
* 使用 OpenAI-compatible endpoint；
* 使用本地推理服务；
* 在模型不可用时切换或降级；
* 查看每次任务实际使用的模型、成本和延迟。

Kin 应支持模型路由，但不应把模型路由变成用户必须理解的复杂配置。

---

### 5.4 一个 Kin，多种化身

从用户视角看，Kin 始终是一个主体，而不是每台设备上各自独立的机器人。

不同客户端拥有不同设备能力：

```text
macOS
- 文件系统
- 剪贴板
- 快捷键
- 菜单栏
- 终端
- 桌面应用联动

iOS / iPadOS
- 分享菜单
- 快捷指令
- 相机和语音
- 通知
- 移动端文件
- 位置等受控权限

Android
- Intent
- 分享入口
- 通知
- 后台任务
- 系统级文件和应用协作

Web
- 通用访问
- 配置管理
- 记忆检查
- 任务审计

CLI
- 开发者工作流
- 脚本和自动化
- 远程运行
- 调试和系统集成
```

设备能力可以不同，但身份、记忆和任务状态应保持连续。

---

### 5.5 简洁优先于功能堆积

Kin 可以拥有复杂的内部系统，但不能把复杂性直接暴露给日常用户。

默认体验应围绕几个稳定入口：

* Ask：询问和讨论；
* Plan：制定计划；
* Do：执行任务；
* Read / Artifacts：阅读与整理会话中的可读交付物（讲义、HTML 资料、长文产物）；
* Remember：管理长期记忆（可从 Artifacts 提炼，但不是同一对象）；
* Review：查看执行结果和历史；
* Routine：管理用户主动创建的长期任务。

**Artifacts** 解决「agent 写完资料 → 下载后丢失上下文 → 多端难读」的交付物问题；**Remember** 解决跨任务、跨模型的可治理长期记忆。二者不要混成一个入口。

高级能力应逐步暴露，而不是一次性展示大量 Agent、Workflow、Skill、Node 和 Provider 概念。

用户可以拥有一个复杂系统，但不应该每天都在管理这个系统。

---

### 5.6 记忆必须可治理

Kin 的记忆不应只是聊天记录或向量数据库。

记忆至少应区分：

```text
Profile Memory
稳定的个人信息和明确偏好

Episodic Memory
发生过的事件、讨论和经历

Semantic Memory
从多次交互中总结出的事实与规律

Procedural Memory
用户喜欢怎样完成某类任务

Task Memory
尚未完成任务的状态和上下文

Relationship Memory
Kin 与用户之间形成的表达习惯、边界和协作模式
```

每一条非原始记忆应尽可能包含：

* 内容；
* 来源；
* 创建时间；
* 更新时间；
* 置信度；
* 适用范围；
* 敏感级别；
* 是否经过用户确认；
* 与其他记忆的冲突关系；
* 过期或复查规则。

Kin 不得把模型推断出的内容伪装成用户明确提供的事实。

用户必须能够查看“Kin 为什么认为这是事实”。

---

### 5.7 Kin 应提供判断，而不是迎合

Kin 的目标不是最大化用户即时满意度。

在用户需要建议时，Kin 应当：

* 理解用户的目标；
* 考虑用户过去的选择和限制；
* 区分事实、推断和意见；
* 提醒明显风险；
* 在有充分理由时提出不同观点；
* 避免仅仅复述用户已经表达的倾向。

Kin 应当站在用户一边，但不意味着永远同意用户。

---

### 5.8 能力逐步授权

Kin 的行为权限应当从低到高逐步开放。

建议采用以下授权层级：

```text
Level 0 — Observe
读取用户主动提供的信息，只进行回答和分析。

Level 1 — Prepare
生成草稿、计划、命令或待执行操作，但不产生外部影响。

Level 2 — Confirmed Action
在每次明确确认后执行邮件、文件、日历、浏览器等操作。

Level 3 — Scoped Delegation
用户为特定工具、时间范围和任务类型授予持续权限。

Level 4 — Routine Automation
执行用户明确创建、可以查看和随时暂停的长期任务。
```

Kin 不应默认获得广泛、永久和不可见的系统权限。

所有具有外部影响的操作应当留下审计记录。

---

### 5.9 可恢复性优先于“全自动”

对于重要任务，Kin 应优先采用可恢复的执行方式：

* 先生成计划；
* 保存任务状态；
* 分步骤执行；
* 支持失败重试；
* 支持暂停和继续；
* 标记已产生的外部影响；
* 在可能时提供撤销或补偿操作。

Agent 执行系统必须假设：

* 模型会犯错；
* 工具会失败；
* 网络会中断；
* 用户会改变决定；
* 设备会离线；
* 外部系统状态会变化。

---

### 5.10 可观测性是产品能力

用户和开发者都应能理解 Kin 做了什么。

一个任务至少应该能够展示：

* 用户目标；
* Kin 的任务计划；
* 使用的模型；
* 使用的工具；
* 关键输入和输出；
* 权限确认；
* 执行时间；
* 成本；
* 错误和重试；
* 最终产生的外部影响。

内部可以存在复杂推理，但产品不能依赖不可解释的黑箱状态来管理真实任务。

---

### 5.11 如无必要勿增实体（范围纪律）

Kin 追求**小而美**，而不是大而全。

在模型与 Agent 能力快速膨胀的时代，更容易跑得快的是：把一件事做到手感正确，而不是先搭一座平台。路线图里可以有远景；**代码与用户可见概念里，没有痛感就不要增加实体。**

#### 纪律

1. **先把主路径做好**：本机上愿意每天打开、记得准、能干活、说得清。
2. **不为遥远架构预支成本**：多设备与编排类能力，等真实摩擦出现再实现。
3. **外部执行器是 Tool，不是第二套 Kin**：可调用第三方 coding agent / harness；它们不是第二套身份或 Runtime。
4. **不以组网基础设施为产品中心**：若以后需要远程访问，优先用户自选或已有连通方式，而不是 Kin 自建网络栈。
5. **少一个用户必须理解的概念，就胜一筹**：复杂可以留在内部，默认体验保持短路径。

#### 增实体前的三问

每增加一个模块、守护进程、同步面、设备抽象或产品名词，先回答：

1. **没有它，今天的主路径是否仍然成立？**
2. **它是在解决已经反复出现的摩擦，还是在预支未来故事？**
3. **用户是否必须多理解一个概念，才能完成日常使用？**

若 (1) 为「成立」且 (2) 为「预支」——默认 **不增加**。

#### 当前默认不增加的实体

可作为长期愿景；**在真实使用反复要求之前，不是默认 v1 表面**：

* 以多设备编排为中心的产品形态
* 单机尚未做好之前的完整同步栈
* 默认开启的桌面控制 / Computer Use
* 面向用户的多 Agent 阵容
* 插件市场与自动路由中台
* 官方云作为身份、记忆或执行的唯一路径

在某项能力成为日常刚需之前，优先更简单的路径：在已经具备工具的那台设备上使用同一个 Kin。

#### 小，但不能空

「少实体」不是做成无差异的聊天壳。以下种子必须保留，但都应**做小**：

| 必留 | 小做法 |
|------|--------|
| 用户拥有 | 本地为主；能导出；官方账户非必须 |
| 模型可换 | 少量 Provider，不搞复杂路由 |
| 记忆可治理 | 确认、查看、删除、来源；宁可少记，不要脏记 |
| 权限可感知 | 外部动作前确认；一条可读的执行时间线 |

#### 一句话

> 做好小的 Kin，再让它慢慢长大。长大必须由使用中的痛感驱动，而不是由完整性焦虑驱动。

## 6. 产品核心组成

### 6.1 Kin Identity

负责维护 Kin 的稳定身份，包括：

* Agent 名称；
* 用户身份；
* 沟通风格；
* 用户设定的行为边界；
* 长期目标；
* 默认决策原则；
* 不同设备和模型之间的一致性。

Identity 不应被写成一个巨大、不可维护的 System Prompt。

它应当由结构化配置、用户明确设置和受治理的长期记忆共同形成。

---

### 6.2 Kin Memory

负责记忆的：

* 提取；
* 存储；
* 检索；
* 更新；
* 合并；
* 冲突发现；
* 过期；
* 删除；
* 导入和导出。

Memory 系统必须将以下内容分开：

* 原始记录；
* 用户明确事实；
* 模型总结；
* 模型推断；
* 用户修正后的结论。

---

### 6.2.1 Kin Artifacts（会话产物，近端能力）

Artifacts 是 agent 会话中产生的**可读交付物**（如主题学习 Markdown、HTML 讲义、长文说明），不是向量记忆，也不是收藏夹书签。

负责：

* 从会话中捕获或由用户保存产物；
* 以本地文件为真源，并建立可检索索引；
* 保留与源任务 / 会话的关联；
* 在控制台中提供库列表与阅读器；
* （后续）基于当前文档的陪读，而不是另起炉灶的无关聊天；
* 可选地**提议**将要点提炼进 Memory / Wiki——默认仍须用户确认。

约束：

* 高影响写入走提议 / 确认，不静默永久化；
* HTML 等富内容按不可信内容沙箱渲染；
* 多端阅读优先复用远程访问梯子访问**同一 daemon**，而不是 Kin 托管网盘；
* 导出与离开时，Artifacts 与任务数据一并可由用户带走。

---

### 6.3 Kin Runtime

Runtime 是 Agent 执行核心，负责：

* 构造上下文；
* 调用模型；
* 调用工具；
* 维护 Agent loop；
* 执行任务计划；
* 管理暂停、恢复和重试；
* 管理人工确认；
* 记录执行轨迹。

Runtime 可以支持子 Agent，但对用户而言，子 Agent 是 Kin 的内部执行方式，而不是一组需要日常管理的独立人格。

Kin 是单一主体，内部可以拥有多个临时 Worker。

---

### 6.4 Provider Layer

Provider Layer 负责屏蔽不同模型服务之间的差异。

核心要求：

* Provider 可插拔；
* Endpoint 可配置；
* API Key 由用户控制；
* 支持本地模型；
* 支持模型能力探测；
* 支持超时、重试和备用模型；
* 记录成本和调用元数据；
* 不将敏感上下文默认发送给所有 Provider。

---

### 6.5 Tool and Skill Layer

Tool 表示一项原子能力，例如读取文件、搜索邮件或创建日历事件。

Skill 表示一个可复用的任务方法，例如：

* 准备会议；
* 总结一天；
* 整理下载文件；
* 分析代码仓库；
* 制定旅行计划；
* 跟踪长期项目。

Tool 和 Skill 应当具备：

* 明确输入输出；
* 权限声明；
* 风险级别；
* 可测试性；
* 版本信息；
* 运行日志；
* 可禁用和卸载；
* 尽可能明确的失败语义。

Kin 应兼容开放工具协议，例如 MCP，但不应让内部架构完全依赖某一种外部协议。

---

### 6.6 Sync Layer

同步系统负责在多个设备之间保持：

* 身份；
* 配置；
* 对话；
* 记忆；
* 任务；
* 工作流；
* 非敏感工具配置；

的一致性。

同步必须考虑：

* 离线编辑；
* 冲突；
* 部分同步；
* 设备撤销；
* 端到端加密；
* 数据版本；
* 模式升级；
* 大附件；
* 敏感凭据本地化。

同步服务必须是可替换组件，而不是 Kin 的中心。

---

### 6.7 Client Shell

每个客户端都应提供一致的核心体验，同时保留平台原生能力。

一致部分包括：

* 同一个 Kin；
* 同一套记忆；
* 同一任务状态；
* 同一权限体系；
* 同一 Provider 和 Skill 配置。

平台差异部分包括：

* 系统集成；
* 输入方式；
* 后台能力；
* 通知；
* 可调用工具；
* 界面布局；
* 设备权限。

---

### 6.8 Trust and Audit

Trust Layer 负责：

* 权限；
* 确认；
* 凭据；
* 沙箱；
* 审计；
* 风险分级；
* 数据出站提示；
* 敏感信息处理；
* 撤销和设备管理。

安全不能作为后期附加模块，而必须贯穿所有架构设计。

---

## 7. 核心交互模型

Kin 的主要交互不应只有聊天。

### Converse

用于讨论、理解、解释和建议。

### Plan

Kin 将复杂目标转化为：

* 步骤；
* 依赖；
* 风险；
* 所需权限；
* 预计使用的工具；
* 需要用户决定的问题。

### Act

Kin 根据计划执行具体任务。

执行前应明确：

* 会影响什么；
* 将使用哪些工具；
* 哪些步骤需要确认；
* 哪些数据会发送到外部服务。

### Review

用户可以检查：

* Kin 完成了什么；
* 哪些步骤失败；
* 哪些决定由 Kin 做出；
* 哪些信息被保存为记忆；
* 是否需要修正或撤销。

### Routine

用户可以将稳定任务保存为长期例程。

Routine 必须具备：

* 明确触发条件；
* 明确执行边界；
* 明确权限；
* 最后运行状态；
* 运行历史；
* 暂停和删除入口。

---

## 8. Kin 的个性设计

Kin 不应预设一个强烈、不可改变的固定人格。

默认个性应当是：

* 清晰；
* 可靠；
* 尊重用户；
* 不过度热情；
* 不故作人类；
* 不刻意卖萌；
* 不轻易承诺；
* 区分事实与判断；
* 能够直接指出问题。

用户可以逐渐调整：

* 回答长度；
* 表达风格；
* 主动程度；
* 风险偏好；
* 建议方式；
* 是否经常提出反对意见；
* 何时主动提醒；
* 哪些事情不应干预。

这些偏好应该是结构化、可查看的配置，而不是隐藏在无法维护的 Prompt 中。

---

## 9. 隐私与安全底线

Kin 必须遵守以下底线：

1. 不在用户不知情的情况下上传私人数据。
2. 不将长期记忆默认用于模型训练。
3. 不在日志中明文记录密钥和敏感凭据。
4. 不把凭据直接暴露给模型。
5. 不默认同步设备级密钥。
6. 不允许插件绕过统一权限系统。
7. 不允许 Agent 隐藏其已执行的外部操作。
8. 不在缺乏明确授权时发送消息、支付、发布内容或删除数据。
9. 用户删除数据后，应清楚说明删除范围和残留备份策略。
10. 用户必须可以完全导出并离开 Kin。

对于高风险操作，应采用确定性策略和工具级约束，而不能只依赖模型遵守自然语言提示。

---

## 10. 开源原则

Kin 的开源版本不是商业版本的演示壳，而应当是一个真正可独立使用的完整个人 Agent。

开源核心应至少包括：

* Agent Runtime；
* Provider 接口；
* 本地记忆系统；
* 权限和审计框架；
* Tool/Skill SDK；
* 基础客户端；
* 数据导入导出；
* 本地使用模式；
* 自托管文档；
* 插件开发文档。

建议优先采用宽松、适合生态扩展的许可证，例如 Apache-2.0。

任何官方托管服务都应当提供便利性，而不能成为以下能力的唯一来源：

* 身份；
* 长期记忆；
* 模型访问；
* 数据同步协议；
* 插件安装；
* 数据导出。

项目应鼓励：

* 第三方客户端；
* 第三方 Provider；
* 第三方同步后端；
* 第三方 Tool 和 Skill；
* 第三方本地模型；
* 社区兼容实现。

---

## 11. 非目标

Kin 初期不试图成为：

* 企业级多租户 Agent 平台；
* 大规模 Agent 集群调度系统；
* 新的大语言模型；
* 完整操作系统；
* 社交网络；
* 虚拟人或数字形象平台；
* 以情感依赖为核心的陪伴机器人；
* 默认全自动控制用户设备的系统；
* 所有 SaaS 产品的统一低代码替代品；
* 单纯的 OpenAI、Claude 或其他模型聊天壳。

Kin 可以使用多 Agent 技术，但不以向用户展示大量 Agent 为目标。

Kin 可以支持云端运行，但不以云端托管为产品中心。

Kin 可以提供陪伴感，但不能通过诱导依赖来提高使用时长。

## 12. MVP 范围

MVP 是 **agent 控制台**：在任何设备上派发、监控、批准外部 agent 任务——self-hosted、跨 agent、不经厂商中继。第一阶段的目标不是完整数字分身，而是第一片每天都值得打开的功能。

### MVP 必须具备

1. 可被桌面壳与 CLI 共用的 Kin daemon（Core）。
2. Agent 适配器：Claude Code 与 Codex 为一等公民；通用 PTY 兜底任意 CLI。
3. 任务生命周期：派发、流式进度、取消、历史。
4. 确认收件箱：桌面与手机上批准权限请求；每次决定都有审计记录。
5. 费用透明：按任务、按 Provider 的 token 与花费。
6. 远程访问梯子：局域网 → 内嵌 tailnet / Funnel → 完整 tailnet；绝不做 Kin 云。
7. 本地任务存储；完整的数据导入和导出。
8. 稳定的桌面客户端（壳）+ 任意设备 Web 控制台。
9. 不需要任何 Kin 账户也能使用核心功能。

### MVP 推荐优先顺序

```text
第一优先级
- 适配器
- 任务派发与监控
- 权限确认
- 执行审计
- 局域网访问

第二优先级
- 远程梯子（tailnet / Funnel）
- 费用统计
- 推送通知
- 更多适配器
- CLI 打磨

第二优先级（近端，控制台之后 / 记忆之前）
- Artifacts 产物库 P0：捕获、本地库、阅读器、源任务关联
- Artifacts 陪读 P1：阅读器侧栏、按文档续聊、标签与检索
  （详见 docs/TODO.md · ADR 0003）

第三优先级
- 可治理记忆 + Profile（v2 故事；可从 Artifacts 提炼）
- 原生移动 App
- 长期 Routine
- 自动模型路由
- 子 Agent
```

首个版本应优先服务开发者本人。

先让 Kin 真正融入创建者自己的日常工作，再扩展到更多用户。

范围扩张时遵守 §5.11：优先加深第一优先级的手感，而不是提前打通第三优先级的版图。Artifacts 先于完整 Wiki/Memory，是因为「可读交付物落地与多端阅读」的痛感更具体，且不要求先建完整记忆治理。

*2026-07 修订：控制台先于对话/记忆交付。对话已经发生在 Kin 监督的 agent 内部；未被满足的是「没有厂商居中的跨 agent 监督」。可治理记忆仍是 v2 的差异化核心。*
*2026-07-17 修订：在第二与第三优先级之间明确 Artifacts（产物库 + 阅读器/陪读）近端主题；Remember 仍为 v2，不与 Artifacts 混名。*

## 13. 跨端路线

核心协议从第一天保持平台中立，客户端按照真实使用价值逐步实现。

### Phase 1：桌面 daemon + 控制台

目标：包住 agent，在本机与局域网内监控和批准。

### Phase 2：远程可达

目标：访问梯子（tailnet / Funnel）、手机 Web 控制台、推送通知。

### Phase 3：原生伴随 App 与更深的多设备

目标：同一 API 上的原生移动客户端；身份与记忆在明确授权下跨设备延续（v2）。

## 14. 技术决策过滤器

每一个重要功能或架构方案，都应回答以下问题。

### 用户所有权

* 它是否增加了厂商绑定？
* 用户是否可以导出和迁移？
* 没有官方服务器时是否仍然可用？

### 连续性

* 更换模型后是否仍然工作？
* 更换设备后是否仍然是同一个 Kin？
* 升级后是否保留历史和配置？

### 简洁性

* 这是用户必须理解的概念吗？
* 能否通过可靠默认值隐藏复杂性？
* 是否在为少数高级场景破坏主要体验？

### 可解释性

* 用户能否知道 Kin 为什么这样做？
* 用户能否检查它使用了什么信息？
* 用户能否修正错误状态？

### 安全性

* 失败时最坏结果是什么？
* 操作是否可以撤销？
* 权限是否足够小？
* 模型是否接触了不必要的数据？

### 开放性

* 第三方是否可以实现替代组件？
* 接口是否有清晰规范？
* 是否存在未经说明的私有依赖？

如果一个方案明显违背其中多项原则，它就不应进入 Kin Core。

---

## 15. 成功标准

Kin 的成功不能只用下载量、消息数或 Token 消耗衡量。

更重要的指标包括：

* 用户是否持续使用 Kin 处理真实事情；
* 用户切换模型时是否无需重新配置个人上下文；
* 用户是否愿意将自己的习惯沉淀为 Skill 和 Routine；
* Kin 使用长期记忆后是否减少重复解释；
* 记忆被用户修正的频率是否逐渐下降；
* agent 生成的学习资料 / 长文是否无需流落下载文件夹即可多端阅读，并保留与源会话的关联；
* 工具执行成功率；
* 任务失败后的恢复率；
* 用户能够理解执行轨迹的比例；
* 用户是否可以在不依赖官方云服务的情况下长期使用；
* 社区是否能够独立开发 Provider、Tool、Skill 和客户端。

最核心的产品验证是：

> 当用户更换设备或模型时，他仍然认为面前的是同一个 Kin。

## 16. 产品语气

Kin 的对外表达应当：

* 克制；
* 清晰；
* 有工程可信度；
* 不夸大“自主智能”；
* 不宣传虚假的完全自动化；
* 不使用“替代人类”叙事；
* 强调所有权、连续性、开放性和实际能力。

推荐的产品文案：

> Kin is a local-first personal agent that carries your memory, habits, and capabilities across devices and models.

更简洁的版本：

> One Kin. Every device. Any model.

仓库简介可以写为：

> An open-source, local-first personal agent that keeps your memory, habits, tools, and identity independent from any model provider.

中文简介：

> Kin 是一个开源、本地优先、跨设备、模型无关的个人 Agent。它保留你的记忆、习惯、工具和身份，不依附于任何模型厂商。

## 17. 项目宣言

Kin 不属于某一个模型。

Kin 不属于某一个云平台。

Kin 不因为用户更换设备而重新开始。

Kin 不会在用户看不到的地方形成不可纠正的记忆。

Kin 不会把更多权限误认为更强的智能。

Kin 可以越来越了解用户，但用户始终拥有解释权、修改权和最终决定权。

Kin 的价值不在于它像人，而在于它能够长期、可靠、可控地成为用户能力的一部分。

> Kin should grow with the user, without owning the user.
